## 경위서 (사관 기록)

### 제1장 — 패배의 기록: Azure Container Apps의 한계

去る 3월 10일 이래, Teams 봇 서비스를 Azure Container Apps에 배포하여 운용코자 하였으나, 그 도정에서 마주한 난관이 심히 중대하여 이를 기록한다.

**첫째, 리전의 족쇄.** Azure for Students 구독은 koreacentral 리전을 허락하지 아니하였다. 부득이 eastasia(홍콩)에 컨테이너를 배치하였으나, 한국 소재 예약 사이트(app.mile.im)와의 네트워크 지연이 Playwright 스크래핑의 타이밍을 흐트러뜨렸다.

**둘째, 자원의 빈곤.** 1 vCPU, 2GB 메모리로 Chromium 브라우저와 Node.js 서버를 동시에 구동하니, 메모리 압박으로 브라우저가 수시로 비정상 종료하였다. `--single-process` 플래그를 붙이면 크래시가 나고, 떼면 메모리가 부족하니 진퇴양난이었다.

**셋째, 유령 같은 성공.** 가장 통탄할 일은 서버가 "예약 확정" 응답을 보내놓고 실제 사이트에는 예약이 생성되지 않는 현상이었다. 로컬에서는 E2E 전체 플로우(로그인→조회→예약→확인→취소)가 완벽히 통과하건만, 서버에서는 폼 제출 버튼 클릭이 실제로 발화하지 않았다. headless Linux 환경과 GUI Windows 환경의 렌더링 타이밍 차이가 원인이었으나, 이를 원격에서 디버깅하는 것은 안개 속에서 바늘을 찾는 것과 다름없었다.

**넷째, 디버깅의 고역.** 코드 한 줄을 고치면 push → CI/CD 빌드(3분) → 배포(2분) → Teams 테스트 → Azure Cloud Shell 로그 확인이라는 10분 이상의 순환을 거쳐야 했다. 이를 10건 이상의 커밋에 걸쳐 반복하니, 하루가 통째로 소진되었다.

**다섯째, 인증의 미궁.** Azure Bot Service는 AppID, AppPassword, TenantID, SingleTenant 설정을 요구하였고, 비밀 만료·불일치 이슈가 반복되어 매번 Azure Portal을 헤매야 했다.

이상의 사유로, 3월 26일자로 Azure Container Apps 배포를 포기하고 Mattermost 전환을 결정한 바 있었다.

### 제2장 — 전환점: "봇이 꼭 서버에 있어야 하는가?"

3월 27일, 근본적 질문이 제기되었다. Teams는 클라이언트 앱이고, Azure Bot Service는 메시지를 중계할 뿐이니, **봇 서버가 Azure에 있어야 할 이유가 없다.** Messaging Endpoint만 도달 가능하면 봇은 어디서든 실행할 수 있다.

이에 착안하여 다음 구조를 설계하였다:

```
Teams 앱 → Azure Bot Service(무료, 라우터) → Cloudflare Tunnel(무료) → 사무실 PC(봇+Playwright)
```

### 제3장 — 사무실 PC 사양 심사

봇을 올릴 기기의 역량을 심사하였다:

| 항목 | 사양 | 판정 |
|------|------|------|
| CPU | AMD Ryzen 5 5600G (6코어/12스레드) | Azure 대비 6배 |
| RAM | 32GB DDR4 (16GB x 2) | Azure 대비 16배 |
| 디스크 | WD Blue SN570 250GB NVMe | 충분 |
| OS | Windows 11 Home | 로컬 E2E 성공 환경과 동일 |
| 네트워크 | 한국 직결 | 홍콩 우회 해소 |

봇 서버 + Playwright의 실 사용량은 CPU 한자릿수 %, RAM 약 500MB로, 32GB 중 1.5%에 불과하다. 여타 업무와 병행하여도 전혀 부담이 없는 수준이다.

### 제4장 — Cloudflare Tunnel 구축

#### 4-1. cloudflared 설치

GitHub Releases에서 `cloudflared-windows-amd64.exe`(63MB)를 직접 다운로드하여 `C:\Users\sokch\cloudflared.exe`에 배치하였다. winget이 PATH에 없어 직접 다운로드 방식을 택하였다.

#### 4-2. 봇 서버 기동

```bash
npm run build          # TypeScript 컴파일
npm start              # localhost:3978 기동
curl localhost:3978/health  # → {"status":"ok"}
```

빌드 시 `dist/bot/cards/search.json` 누락 오류가 발생하여, 카드 JSON 파일을 수동 복사하는 단계를 추가하였다.

#### 4-3. Cloudflare Tunnel 기동

```bash
cloudflared.exe tunnel --url http://localhost:3978
```

결과:
- **터널 URL**: `https://russell-occurring-challenges-colorado.trycloudflare.com`
- **접속 위치**: `icn01` (인천, 한국) — 한국 엣지 노드에 자동 연결됨
- **프로토콜**: QUIC
- **외부 health check**: `GET /health` → `{"status":"ok"}` 확인

#### 4-4. 시작 스크립트 작성

`scripts/start-local.sh` — 빌드, 서버 기동, 터널 기동, health check를 일괄 수행하는 스크립트를 작성하였다. `Ctrl+C`로 양쪽 프로세스를 동시에 종료하는 cleanup 핸들러를 포함한다.

### 제5장 — Azure 엔드포인트 변경 및 Teams 앱 패키지

#### 5-1. Messaging Endpoint 변경

Azure Portal → Bot Services → cg-teams-booking → Settings → Configuration에서 Messaging Endpoint를 변경 완료:

```
[이전] https://<container-app>.azurecontainerapps.io/api/messages
[변경] https://russell-occurring-challenges-colorado.trycloudflare.com/api/messages
```

#### 5-2. Teams 앱 패키지 생성

Teams에서 봇과 대화하려면 앱 매니페스트 패키지가 필요하여 `appPackage/` 디렉토리를 생성:

| 파일 | 설명 |
|------|------|
| `appPackage/manifest.json` | Teams 앱 매니페스트 v1.17 (봇 명령어: 예약, 현황, 취소, 도움말) |
| `appPackage/color.png` | 192x192 앱 아이콘 (꿀벌 캐릭터) |
| `appPackage/outline.png` | 32x32 아웃라인 아이콘 |
| `cg-teams-booking.zip` | 위 3개 파일을 묶은 Teams 업로드용 패키지 |

#### 5-3. Teams 앱 업로드 — 관리자 승인 필요

사용자 지정 앱 업로드 시도 시 **관리자 승인 요청** 화면이 표시됨. Teams 관리자가 Teams 관리 센터(admin.teams.microsoft.com)에서 앱을 승인해야 조직 내 사용 가능. 관리자 확인 필요.

### 제6장 — 남은 과업

#### 고정 터널 설정 (권장)

현재 Quick Tunnel은 재시작 시 URL이 변경된다. 안정 운용을 위해 Cloudflare 계정으로 Named Tunnel을 생성하면 고정 도메인을 받을 수 있다:

```bash
cloudflared.exe tunnel login                    # Cloudflare 계정 인증
cloudflared.exe tunnel create teams-bot         # 고정 터널 생성
cloudflared.exe tunnel route dns teams-bot teams-bot.yourdomain.com  # DNS 등록
cloudflared.exe tunnel run teams-bot            # 실행 (URL 불변)
```

이 경우 Azure Bot Service의 Endpoint를 한 번만 설정하면 이후 변경 불요.

#### Windows 서비스 등록 (선택)

PC 부팅 시 자동 시작을 원할 경우:

```bash
cloudflared.exe service install   # Windows 서비스로 등록
```

Node.js 서버는 `pm2` 또는 `node-windows`로 서비스화 가능.

## 기술적 결론

Azure는 이 프로젝트의 규모와 성격에 과한 인프라였다. 사용자 수 명, 일 예약 수십 건의 사내 봇에 컨테이너 오케스트레이션, CI/CD 파이프라인, 클라우드 리전 선택을 고민할 필요가 없었다. 사무실 PC 한 대와 무료 터널이면 충분하고도 남는다.

로컬에서 되는 것이 서버에서 안 되는 문제 — 이 괴리를 해소하는 가장 확실한 방법은 **로컬을 서버로 만드는 것**이었다.
